### 배포 환경에서 DB(스키마/마스터·레퍼런스 데이터) 구성 방법 정리

> 전제: `Spring Boot + JPA + MySQL`  
> 목표: 로컬에서 개발한 뒤 AWS(EC2)로 배포해도 **같은 스키마 + 같은 마스터/레퍼런스 데이터(아이템/주술/매핑)** 를 안전하게 재현

---

### 1) 결론(추천 아키텍처)
- **운영 DB는 로컬 DB를 “원격으로 그대로” 쓰지 않는다**(보안/가용성/지연 문제).
- 운영에서는 아래 둘 중 하나를 선택:
  - **(추천)** `AWS RDS(MySQL)` + EC2(애플리케이션)
  - `EC2` 인스턴스 내부에 `MySQL` 직접 설치(운영/백업 난이도↑)
- 스키마/마스터·레퍼런스 데이터 “초기 적재(시드)”는:
  - **(추천)** `Flyway`(또는 Liquibase)로 **마이그레이션 + 시드(seed) 데이터**를 버전 관리
  - 보조 수단으로 “초기 1회” 로컬 데이터 **덤프/복원**(필요 시)

---

### 2) 왜 “로컬 DB를 배포 후 그대로” 쓰기 어렵나?
로컬 DB를 EC2에서 그대로 쓰려면 로컬 PC가 운영 DB가 되는 구조인데, 현실적으로 문제가 큼:
- **가용성**: 내 PC 꺼지면 서비스 DB도 다운
- **보안**: DB 포트 외부 오픈/계정 관리/침입 리스크
- **네트워크**: 지연/끊김/공인IP 변동

따라서 운영 DB는 AWS에 따로 두고, 데이터는 **이관(덤프/복원)** 또는 **시드(seed) 스크립트**로 넣는 게 정석.

---

### 3) 운영에서 “마스터/레퍼런스 데이터”를 넣는 대표 방식 3가지

#### 3.1 Flyway 마이그레이션으로 넣기 (가장 추천)
**장점**
- 스키마 변경 + 데이터 변경을 **버전으로 추적**
- 배포 환경이 바뀌어도 “실행만 하면 동일 상태”를 재현
- 팀 협업에 강함(누가 언제 무엇을 바꿨는지 기록)

**운영 패턴**
- `V1__init.sql`: 테이블 생성
- `V2__seed_items.sql`: 아이템/세트/주술/적용매핑(마스터/레퍼런스 데이터) 초기 적재(시드)
- 필요 시:
  - `V3__add_new_items_2026_03.sql` 처럼 “추가/수정”을 점진적으로 누적

**중요 포인트(마스터/레퍼런스 데이터 설계 팁)**
- 아이템/주술처럼 “사전 데이터”는 가능하면:
  - **자연키(예: code)** 를 두고, 삽입은 `INSERT ... ON DUPLICATE KEY UPDATE` 같은 **업서트 전략**을 고려
  - 단, Flyway는 “한 번 실행된 버전 스크립트는 수정하지 않는” 게 원칙 → 수정이 필요하면 새 버전을 추가
- 대량 데이터면:
  - 초기만 넣고 이후에는 “변경분만” 누적 마이그레이션으로 추가

#### 3.2 SQL 시드 파일을 별도 실행
**장점**
- 단순함(운영에서 SQL 파일 한번 실행)
**단점**
- 버전 관리/재현성/자동화가 Flyway보다 약함(사람이 누락하기 쉬움)

#### 3.3 애플리케이션 시더(부팅 시 데이터 넣기)
**장점**
- 실행만 하면 자동 주입 가능
**단점**
- 운영에서 “중복 삽입/버전 관리/부분 실패”를 설계하지 않으면 사고가 나기 쉬움  
→ 운영에서는 Flyway 같은 마이그레이션 도구가 보통 더 안전.

---

### 4) “로컬 DB 데이터를 AWS DB로 옮기는” 방법(초기 1회 이관)
마스터/레퍼런스 데이터를 이미 로컬 DB에 많이 쌓아둔 경우, 초기 1회는 **덤프/복원**이 가장 빠름.

#### 4.1 MySQL 덤프/복원 흐름(개념)
- 로컬에서:
  - 스키마+데이터 덤프 생성
- AWS(RDS/EC2 MySQL)에서:
  - 덤프 파일 import

**주의**
- 운영에서는 “덤프/복원”은 **초기 1회**로 제한하고, 이후부터는 Flyway로 변경분을 관리하는 게 깔끔함.
- 로컬/운영의 charset/collation, time_zone, sql_mode가 다르면 이슈가 날 수 있어 사전 통일 권장.

---

### 5) 추천 운영 절차(실전 체크리스트)

#### 5.1 DB 준비
- **RDS(MySQL)** 생성 (권장)
- 보안 그룹:
  - DB는 **EC2에서만 접근 허용**(0.0.0.0/0 금지)

#### 5.2 애플리케이션 설정
- 배포 환경에서 `spring.datasource.url/username/password` 를 **환경변수/Secrets**로 주입
- `ddl-auto`는 운영에서 보통 `validate` 또는 `none` 권장  
  (스키마 생성/변경은 Flyway가 담당)

#### 5.3 Flyway 적용
- 배포 시 애플리케이션이 시작되면서:
  - (1) Flyway가 미적용 마이그레이션을 순서대로 적용
  - (2) 마스터/레퍼런스 데이터(아이템/주술/적용 매핑)가 초기 적재(시드)됨

#### 5.4 데이터 업데이트 운영
- “아이템/주술 추가” 같은 사전 데이터 변경이 생기면:
  - 새 마이그레이션 파일(`Vx__...sql`)로 추가
  - 운영/스테이징/로컬 모두 동일하게 반영

### 6) 이 프로젝트에서 “마스터/레퍼런스 데이터”로 관리해야 할 것(예시)
`docs/entity-model.ko.md` 기준, 최소 아래는 마스터/레퍼런스 데이터 성격:
- `Item`(재료/장비 정의)
- `EquipmentItem`(슬롯/외변/일반/세트 소속)
- `EquipmentSet`, `EquipmentSetPiece`
- `Ritual`(주술 정의: 성공/대성공 표기 포함)
- `RitualApplicability`(주술 ↔ 장비 적용 가능 매핑)

### 7) 추천 결론(한 줄)
운영에서는 **AWS(RDS/EC2) DB를 별도로 두고**, 스키마/마스터·레퍼런스 데이터는 **Flyway로 “버전 관리되는 방식”으로 초기 적재(시드)** 하는 게 가장 안전하고 유지보수하기 쉽다.
