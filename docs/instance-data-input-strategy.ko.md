### 인스턴스(행) 데이터 입력/운영 전략 제안 (코드 구현 없음)

> 목적: “거상 거래 서비스”에서 **오픈소스/공식 데이터가 없어서 사람이 직접 만들어야 하는 데이터(행 데이터)** 를  
> 어떻게 **안전하고 반복 가능**하게 입력/수정/배포할지에 대한 **기획 레퍼런스**를 남긴다.  
> 전제: 지금은 기획 단계이며 **코드 구현은 나중에** 한다.  
> 제약: **`items` 관련 코드는 구현하지 않는다**(엔티티/CRUD/시드/어드민 등).

---

### 1) 먼저 구분: “정의(마스터/레퍼런스)” vs “운영(트랜잭션)”
`docs/entity-model.ko.md`의 레이어를 기준으로, 데이터는 아래처럼 성격이 다르다.

- **Catalog(정의/사전 데이터)**
  - 예: 세트 정의, 주술 정의, “어떤 장비에 어떤 주술이 가능한지” 매핑 등
  - 특징: 변경 빈도는 상대적으로 낮지만, **정합성(FK/유니크)과 버전 추적**이 중요
- **Listing(등록 데이터) / Trade(거래 확정 데이터)**
  - 사용자 행동으로 쌓이는 데이터(거래글, 신청, 확정, 신고 등)
  - 특징: 입력 주체가 유저/관리자이고 변경/삭제 정책(감사/분쟁)이 중요

> 결론: “사람이 직접 넣어야 한다”는 문제는 보통 **Catalog(정의 데이터)** 쪽에서 크게 발생한다.  
> 운영 데이터는 서비스가 돌아가면 자연스럽게 쌓인다.

---

### 2) 추천 원칙(한 줄 요약)
- **정의 데이터는 “버전 관리되는 시드”로**, 운영 데이터는 **관리자 도구(최소 기능) + 감사로그**로 관리한다.

---

### 3) 입력 경로 옵션 3가지(추천 순)
#### 3.1 (추천) Flyway 시드(DDL/Seed)로 “정의 데이터”를 버전 관리
`docs/db-seed-and-deploy.ko.md`와 일관된 전략.

- 장점
  - 배포 환경이 바뀌어도 **재현 가능**
  - 변경 이력(누가/언제/무엇을) 추적이 쉬움
- 운영 방식(권장)
  - “한 번 실행된 마이그레이션 파일은 수정하지 않고, 변경분은 새 버전 파일로 추가”
  - 대량 데이터는 “초기 1회 시드 + 이후 증분(추가/정정) 마이그레이션”

#### 3.2 CSV/엑셀 → Import(대량 입력) “최소 백오피스”
정의 데이터가 사람이 입력해야 한다면, **CRUD UI를 먼저 만드는 것보다 Import가 비용 대비 효과가 좋다.**

- 권장 최소 스펙(기획)
  - CSV 업로드 → **검증(pre-validate)** → 미리보기(적용 건수) → 실행(Insert/Upsert) → 결과 리포트
  - 실패 행 다운로드(에러 사유 포함)
  - Upsert 기준키(자연키/코드 등) 설계가 핵심

#### 3.3 완전 수동 CRUD 관리자 페이지(마지막에)
“화면에서 한 행씩 추가/수정”은 **초기 구축은 쉬워도 대량 입력엔 비효율**일 때가 많다.

- 추천 사용처
  - 빈번한 핫픽스/소수 행 수정
  - 운영 중 예외 케이스를 빠르게 처리해야 하는 테이블

---

### 4) “MySQL Workbench로 가져오기”는 어디까지 가능?
- Workbench의 **Reverse Engineer**는 기본적으로 **테이블 정의(스키마)와 제약**을 읽어와 모델(EER)로 만든다.
- **행 데이터(인스턴스)를 자동으로 모델에 담아 가져오진 않는다.**
- 인스턴스 데이터는 별도 **Export/덤프(mysqldump 등)** 로 내보내고, Import로 넣는 흐름이 필요하다.

---

### 5) 관리자(백오피스)를 만든다면: 최소 기능 제안(기획)
“관리자 페이지를 만드는 게 좋냐?”는 **데이터가 자주 바뀌는지 / 입력자가 여러 명인지 / 검증·감사가 필요한지**에 따라 달라진다.

#### 5.1 만들면 좋은 경우
- 데이터가 **주기적으로 업데이트**됨(밸런스 패치/운영 정책/표기 규칙 변경 등)
- 입력자가 2명 이상(기획/운영/GM)
- 실수가 치명적(FK 깨짐, 검색/등록 UX 붕괴)

#### 5.2 최소 백오피스 요구사항(권장)
- **권한**
  - VIEWER / EDITOR / ADMIN (최소 3단계)
- **감사 로그(Audit)**
  - 누가/언제/무엇을(전/후 값) 변경했는지 기록
- **검증**
  - 참조 무결성(FK), 유니크, 범위/형식 체크
- **대량 작업**
  - CSV Import/Export (이게 핵심)
- **롤백/되돌리기(선택)**
  - 최소: “이전 값 복원”이 가능한 변경 이력

---

### 6) `items`를 구현하지 않는다면(현재 제약 반영)
`items`가 실제로 중요한 참조 축이긴 하지만, 당장 구현을 금지한다면 아래 중 하나로 “기획 상 공백”을 메우는 게 안전하다.

- **옵션 A: `items`는 ‘외부에서 주어지는 참조’로 취급**
  - 다른 도메인(예: 주술/세트/등록)에서 “아이템명/코드”를 **문자열로만** 참조하는 방안을 기획에 명시
  - 리스크: 오타/중복/표준화 문제 → Import 단계에서 표준화 룰 필요
- **옵션 B: `items`는 스코프 밖(Out of Scope)으로 명시**
  - MVP에서 “아이템 사전” 없이도 성립하는 플로우(예: 거래글은 자유 텍스트 기반)로 한정
  - 리스크: 검색/필터/시세 정밀도 저하 → 이후 확장 로드맵에 명시

> 어떤 옵션이든, “나중에 `items`를 도입할 때의 마이그레이션(문자열 → 정규화)” 비용이 생길 수 있으니  
> 초기에 표준화 규칙(별칭/정규화 키)을 문서로 남겨두는 걸 권장.

---

### 7) 추천 결론(현 단계에서의 최적 해법)
- **정의 데이터(Catalog)**: Flyway 시드로 버전 관리(초기) + 변화가 잦으면 CSV Import를 곁들인다.
- **운영 데이터(Listing/Trade)**: 관리자 도구는 “대량 입력 + 검증 + 감사로그” 중심으로 최소화한다.
- **`items`**: 현재는 구현하지 않으므로, MVP에서의 참조 방식(자유 텍스트/외부 코드/스코프 제외)을 명확히 문서화한다.
