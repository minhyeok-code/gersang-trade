### 추가기능 구체화 — 능력치별 가성비(속성/속성깎/저항깎)

### 1) 목표
- 사용자가 “가성비 비교” 카테고리에서 **아이템 / 능력치 값 / 전월 평균가 / 가성비(가격÷능력치)** 를 한 눈에 비교
- 핵심 능력치:
  - **속성값**
  - **속성깎(상대 속성값 깎기)**
  - **저항깎(상대 저항 깎기)**

---

### 2) 계산 규칙(제안)

#### 2.1 전월 평균가
- **아이템 가격 = 전월(지난 달) 거래 확정 기준 평균가**
  - 예: 오늘이 3월이면 2월 확정 거래를 집계한 월 평균가 사용
- 거래 건수가 너무 적으면(예: 1~2건) 신뢰도가 낮으므로:
  - **표시하되 “표본 적음” 배지** 또는 최소 거래건수 조건(정책 결정)

#### 2.2 가성비 계산식
- 가성비(낮을수록 좋음):
  - **속성값 가성비** = `전월 평균가 / 속성값`
  - **속성깎 가성비** = `전월 평균가 / 속성깎`
  - **저항깎 가성비** = `전월 평균가 / 저항깎`
- 분모가 0이면:
  - 해당 카테고리에서는 **노출 제외**(또는 “-” 처리)

#### 2.3 능력치 값의 기준(중요)
가성비 비교는 “아이템 정의”의 능력치를 써야 일관됨. 아래 중 하나를 선택해야 함:
- **(권장) ‘아이템 기본 능력치’ 기준**: 마스터 데이터로 저장된 해당 아이템의 기본 속성/깎 수치
- (확장) 옵션/강화(주술 등)까지 포함한 “개체(개별 등록물)” 능력치 기준  
  - 이 경우 변형이 너무 많아져 “아이템 단위 랭킹”이 아니라 “등록물 단위 랭킹”이 됨

> 이번 요구(리스트 카드로 아이템 비교)는 “아이템 단위”가 자연스럽기 때문에 **기본 능력치 기준**을 추천.

---

### 3) 저장해야 할 데이터(엔티티/테이블 관점)

#### 3.1 아이템 능력치(마스터/레퍼런스 데이터)
`ItemStat`(개념)
- **itemId(FK → Item)**
- **statType**: `ELEMENT_VALUE` | `ELEMENT_PIERCE`(속성깎) | `RESIST_PIERCE`(저항깎)
- **element(optional)**: 속성 구분이 필요하면 `FIRE/WATER/WIND/EARTH/LIGHTNING` 등 (게임 정의에 맞춰 확정)
- **value(number)**

표현 방식 2가지 중 택1:
- **A안(정규화, 추천)**: 위처럼 행 단위로 여러 능력치를 저장(확장 쉬움)
- **B안(컬럼형)**: `elementValueTotal`, `elementPierceTotal`, `resistPierceTotal` 같은 합계만 저장(단순하지만 확장 약함)

#### 3.2 전월 평균가(월 집계)
`TradeStatMonthly`(개념, 기존 집계 섹션과 연결)
- **statMonth(YYYY-MM)**
- **statKey = itemId** (가성비 기능 1차 버전은 item 단위 추천)
- **avgPrice**, **tradeCount**, (선택) min/max

> 옵션별 시세까지 가성비에 반영하고 싶어지면  
> `statKey`에 “옵션/주술/세트”를 포함하는 확장안을 별도로 설계.

#### 3.3 (선택) 가성비 랭킹 캐시
조회가 매우 잦다면, 매 요청마다 `avgPrice/스탯` 계산 대신:
- `ValueMetricMonthly`(개념)
  - `month`, `itemId`, `metricType`, `statValue`, `avgPrice`, `valueForMoney(=avgPrice/statValue)`
  - 배치로 한 번에 갱신(월 1회 또는 매일)

### 4) 화면/UX 구체화 — “속성값 가성비 비교” 카테고리

#### 4.1 리스트 카드(필수 표시)
- **아이템명**
- **능력치 값**(카테고리에 따라 속성값/속성깎/저항깎)
- **전월 평균가**
- **가성비** = `전월 평균가 / 능력치 값`
- (추천) **전월 거래건수**(신뢰도 힌트)

#### 4.2 정렬/필터(제안)
- 정렬:
  - 기본: **가성비 오름차순(가장 가성비 좋음)**
  - 보조: 평균가 내림차순, 능력치 내림차순, 거래건수 내림차순
- 필터:
  - 아이템 타입/슬롯(장비면), 세트 여부
  - (element 적용 시) 특정 속성만 보기(예: 화/수/풍/지/번개)
  - 최소 거래건수(예: 5건 이상)

---

### 5) 오픈 이슈(확정 필요)
- **속성 종류(원소) 정의**: 게임 내 속성 종류가 정확히 무엇인지(5원소인지, 이름/코드)
- **능력치 값의 “합산 기준”**:
  - 단일 속성 기준 비교 vs 모든 속성 합계 기준 비교(예: 속성값 총합)
- **전월 평균가 산정 키**:
  - `itemId`만으로 충분한지, “옵션/주술 등”을 분리해야 의미가 있는지

---

### 6) 속성값 데미지 계산기 — 상세 기획

#### 6.1 개요
사용자가 현재 보유한 속성값을 입력하고 구매 후보 아이템을 선택하면,
해당 아이템 구매 시 **속성 데미지 증가분(m%p)** 을 실시간으로 계산해 알려주는 기능.

#### 6.2 게임 메커니즘
- **속성 추가 데미지 공식**: `n% = (3x - y) / 2`
  - `x`: 내 속성값
  - `y`: 몬스터(또는 상대) 속성값
  - `n`: 최솟값 **-50%**, 최댓값 **+50%** (범위 이탈 시 cap 적용)
- 속성값이 높을수록 추가 데미지가 증가하며, 몬스터 속성값이 높으면 오히려 감소할 수 있음
- **저항값**: 상대의 속성 데미지를 경감하는 수치. 현재 이 공식의 직접 변수는 아니지만,
  추후 저항값 관련 계산식이 확정되면 동일한 방식으로 확장 가능

#### 6.3 데미지 증가분 계산 방법
```
n_before = clamp((3 * x       - y) / 2, -50, 50)
n_after  = clamp((3 * (x + δ) - y) / 2, -50, 50)
m        = n_after - n_before   // 단위: %p
```
- `δ` (delta): 구매 후보 아이템의 속성값 (`ItemStat.statType = ELEMENT_VALUE`)
- 예시:
  - x=100, y=0, δ=30 → n_before=150→cap50%, n_after=50%(이미 cap) → m=0%p (의미 없음, 초과 안내 필요)
  - x=30, y=0, δ=20 → n_before=45%, n_after=50%(cap) → m=+5%p
  - x=10, y=0, δ=20 → n_before=15%, n_after=45% → m=+30%p

#### 6.4 필요한 데이터
| 데이터 | 출처 |
|-------|------|
| 아이템의 속성값(δ) | `ItemStat.statType = ELEMENT_VALUE`, `element = 해당 속성` |
| 내 현재 속성값(x) | 사용자 런타임 입력 (저장 불필요) |
| 몬스터 속성값(y) | **결정 필요** — 사용자 입력 / 기준값 제공 / y=0 고정 중 택1 |

> `ItemStat.ELEMENT_VALUE` 필드는 가성비 기능(5절)과 이 계산기 기능에서 **공유**됨.
> 별도 필드 추가 없이 기존 `ItemStat` 구조로 커버 가능.

#### 6.5 UX 흐름(제안)
1. 사용자가 **내 현재 속성값** 입력 (+ 필요 시 몬스터 속성값 입력)
2. 아이템 목록에서 구매 후보 선택 (또는 아이템 상세 페이지 내 계산기 패널)
3. 실시간으로 `m%p` 결과 표시
   - 증가: `이 아이템 구매 시 속성 데미지 +m%p 향상 (→ 총 n_after%)`
   - 이미 cap 도달: `현재 속성값이 이미 최대 데미지(+50%)에 도달해 있습니다`
   - 음수 결과(의미 없음): `이 속성값 구간에서는 효과가 없습니다`

#### 6.6 복수 아이템 비교(확장)
- 후보 아이템 여러 개를 나란히 비교: `아이템A +m1%p / 아이템B +m2%p / 아이템C +m3%p`
- 가성비 기능(가격 ÷ m 값)과 결합하면 “데미지 증가 1%p당 비용” 지표 도출 가능

#### 6.7 오픈 이슈
- **몬스터 속성값(y) 처리 방식**: y=0 고정(단순화) vs 사용자 직접 입력 vs 서비스에서 대표 몬스터 프리셋 제공
- **저항값(resistValue) 공식 확정**: 저항값에 대한 공식이 추가되면 동일 패턴으로 확장 예정
- **속성 종류별 분리**: 화/수/풍/지/번개 각 속성값을 개별 입력받아 속성별 계산기로 확장할지 여부
